RL Infra Learning Thesis

Six production-grade RL post-training frameworks — Miles, SLIME, SGLang, SkyRL, cosmos-rl, RLinf — read through one shared lens. Underneath the marketing, they all solve the same four problems in different orders. This page is the connection map.

The thesis. Every RL post-training framework is a particular way of stitching together three pillars — a training engine, an inference engine, and an orchestrator — around one cycle (generate → score → train → sync weights). Frameworks differ on five axes: which backends they pick, which algorithms they implement, which domain they target, where the rollout lives, and what numerical guarantees they make. Once you see the cycle and the axes, every repo becomes legible.

The universal architecture

If you read all six repos at the same time, you see the same skeleton in each one. The names differ; the shape doesn't.

① Training engine
Computes gradients, holds the optimizer, owns the policy weights of record.
FSDP · Megatron-LM · DeepSpeed
② Inference engine
Runs the policy to generate rollouts. Optimized for throughput, prefix caching, weight hot-swap.
SGLang · vLLM · TRT-LLM · diffusers
③ Orchestrator
Coordinates workers across GPUs/nodes — placement, weight sync, fault recovery.
Ray · custom NCCL · torchrun
prompts → [② generate] → samples + logprobs → [reward / filter] → batches → [① train step] → new weights → [③ sync] → [② update_weights] → ...

Around these three pillars sits a shared data layer (rollout buffer, reward hub, filter hub) and an algorithm layer (GRPO is the universal default; PPO, DAPO, OPD, Reinforce++ are variants). Once you can locate these six pieces in a repo, you can read its train.py in fifteen minutes.

The elegance of rollout design

Drawn from Chenyang Zhao's Awesome-ML-SYS-Tutorial — the canonical Chinese-language deep dive on SGLang internals and RL system design. The thesis below is what makes modern RL infra 精美 (elegant): not minimalism, but correctness by construction.

Chenyang's core claim. "A large number of RL conclusions derived from papers are based on RL infrastructure that may be extremely flawed." Infrastructure correctness precedes algorithmic speed. Every elegant primitive on this page exists because the alternative would silently corrupt your training run — not because it ran 5% faster.

① The opposing-preferences problem

Training and rollout pull in opposite directions on the same hardware. This is the root tension every RL framework has to resolve.

ConcernTraining prefersRollout prefers
ParallelismHigh TP / PP — split the modelHigh DP — split the batch
MemoryOptimizer states + gradients + activationsWeights + KV cache + CUDA graphs
Throughput unitTokens/step (compute-bound)Trajectories/sec (memory-bound)
GoalUpdate one global policyGenerate N independent samples

If you force identical parallelization, one side bottlenecks. If you give them different parallelization, you need expensive parameter resharding. Every elegant primitive that follows is a way out of this trap.

② The hybrid engine: colocate, serialize, swap memory

verl's answer (and now the dominant pattern across Miles, SLIME, SkyRL, RLinf): put training and rollout on the same GPUs, run them serially, and swap memory ownership between them. This is what 训推一体 (integrated train-inference) actually means.

[T=0] Inference engine holds weights + KV cache + CUDA graphs (~80GB).
      Trainer state is offloaded to CPU/NVMe.

[T=1] release_memory_occupation()
      → KV cache freed, CUDA graphs released, weights pinned.

[T=2] Trainer materializes optimizer/grads on GPU, runs forward+backward,
      computes new policy weights.

[T=3] update_weights_from_distributed(...)
      → NCCL broadcast from training ranks to inference ranks.

[T=4] resume_memory_occupation()
      → KV pool re-allocated, CUDA graphs rebuilt.

[T=5] Next rollout starts on the new policy. Goto T=0.
Why this is elegant. The GPU is owned by exactly one process at any moment. No racing, no torn reads, no "did the broadcast finish before I called generate?" Two functions — release and resume — encode an entire concurrency contract. The serialization cost is real, but the alternative (lock-free shared memory between two distributed runtimes) has no known correct implementation.

③ Four update_weights_from_* paths, one verb

SGLang exposes four implementations behind one interface. Each maps to a real-world deployment topology — and the framework picks based on where the new weights physically live.

MethodWhen the trainer's new weights are in...Used by
update_weights_from_tensorSame Python process as the engineSingle-node colocated; small experiments
update_weights_from_diskA checkpoint file on shared storageDisaggregated rollout pools that scale independently
update_weights_from_distributedMegatron training ranks across the clusterMiles, SLIME, RLinf — the dominant path
update_weights_from_ipcA sibling CUDA process on the same hostSame-host trainer ↔ server, no NCCL setup cost
Why this is elegant. "Update weights" is a verb, not a topology. The framework writes engine.update_weights_from_*(...) and SGLang internally handles tensor reconstruction, dtype preservation under quantization, and the lock that prevents concurrent generation. Adding a new transport (e.g. RDMA) means adding a fifth method — not rewriting every framework that calls it.

④ NCCL is static; RDMA is dynamic

An underrated design constraint with massive architectural consequences:

  • NCCL process groups fix participants at creation. Adding a new inference node means destroying and recreating the group — every existing rank re-rendezvouses.
  • RDMA's point-to-point model treats each connection independently. A new node establishes fresh Queue Pairs without disturbing existing links.

This single fact explains why colocated frameworks (Miles, SLIME) lean into NCCL via init_weights_update_group, while disaggregated systems (cosmos-rl, SkyRL's remote inference engine) prefer RDMA or disk for weight sync. Elasticity is incompatible with NCCL; if you need to add inference capacity mid-training, the topology forces your hand.

⑤ RadixAttention is "free" because GRPO is group-structured

GRPO generates N completions per prompt and normalizes rewards within the group. This algorithmic choice has a system-level windfall: all N completions share an identical prefix.

prompt P (5000 tokens — system + few-shot + question)
   ├── completion c₁  (decoded with shared P prefix)
   ├── completion c₂  (decoded with shared P prefix)
   ├── completion c₃  (decoded with shared P prefix)
   └── completion c₄  (decoded with shared P prefix)

  Without prefix cache:  4 × prefill(P) + 4 × decode    ← prefill dominates
  With RadixAttention:   1 × prefill(P) + 4 × decode    ← almost-free group rollouts
Why this is elegant. An algorithmic decision (group sampling for variance reduction) and a system optimization (radix-tree KV cache) compose multiplicatively. Neither was designed for the other. The reference-counted radix tree means cache lifetime is automatic — completions release their tail KVs as they finish, the shared prefix lives until the last group member exits. No manual lifecycle management ever.

⑥ Engine vs Server duality

SGLang ships the same logic as two shells. Choose by topology, not feature:

Why this is elegant. Same scheduler, same RadixAttention, same weight-update mixin. Co-located runs save HTTP overhead; disaggregated runs get network locality. The framework above doesn't need to care which shell it has — both honor the same Python signature.

⑦ Partial rollout and the staleness honesty

Strict on-policy RL says: wait for every trajectory to finish under policy θₜ before running the gradient step. In practice this is fatal — long-tail trajectories (think 8K-token reasoning chains) leave the GPU idle for minutes.

Partial rollout says: start training on the completions that finished, let the others continue under (now slightly stale) θₜ. This is mathematically incorrect — until you add importance sampling corrections (Miles' TIS/MIS) or just operationally bound the staleness (Kimi's approach: context-length checkpointing, dynamic batch reordering).

Why this is elegant. The system tells you it's lying about being on-policy. The IS weights are explicit; the staleness is a parameter, not a bug. This is "rational compromise between engineering efficiency and algorithmic rigor" — but the compromise is visible in the code, not hidden behind it.

⑧ The thesis: correctness by construction

"The bitter experience of RL system development teaches that theoretical simplicity often masks implementation complexity."
— Chenyang Zhao

Every primitive on this page exists because the obvious alternative is silently wrong:

  • release_memory_occupation exists because shared-memory two-process GPU access has no correct implementation.
  • The four update_weights_from_* paths exist because one method can't honestly serve four topologies.
  • RadixAttention's reference counting exists because manual lifecycle on a 64-way concurrent KV pool cannot be debugged.
  • The Engine/Server duality exists because forcing HTTP onto colocated runs adds 30%+ latency for no benefit.
  • Importance-sampling corrections exist because partial rollout without them slowly drifts the policy off-distribution.

Modern RL infra is elegant because each primitive makes a class of bugs unrepresentable. Not "easy to avoid" — unrepresentable. The pause-update-resume contract means a torn read can't happen. The reference-counted radix tree means a use-after-free can't happen. The four-path weight-update API means you can't accidentally use a topology your transport doesn't support. This is what 精美 means in this context: correctness as a property of the architecture, not the engineer.

Six design decisions, interrogated

For each decision: why this · why not the alternative · engineering benefit · scaling property.

① 训推一体 (colocate) vs 训推分离 (disaggregate)

② SGLang vs vLLM as the rollout backend

ChoiceCost of the alternative

Critical asymmetry. Inference engines (vLLM, SGLang) use kernel fusion that differs at different batch sizes. For sampled tokens this is fine. For logprobs feeding loss it is not — numerical drift accumulates to unacceptable error. This is "the bug we still haven't fixed since the BERT era." Practical answer: critic and reward use the training engine's forward(), not the inference engine. SGLang is used to generate; logprobs that matter are recomputed on the training side.

③ Radix tree vs hash-based prefix cache

A hash cache hashes the full prefix string. Eviction is all-or-nothing — if "A→B→C→D" is cached and you need "A→B→C", you either keep the long entry (memory wasted) or evict it (cache miss). A radix tree stores "A→B→C" as ancestor nodes and "D" as a child; evicting D leaves A→B→C intact for reuse. Fine-grained eviction with reference counting means streaming multi-turn conversations and long-context RL share prefixes naturally — even when full conversations differ.

④ Four update_weights_from_* paths vs one universal method

The four methods exist because the sharding semantics on the trainer side genuinely differ:

  • FSDP with single TP: just call full_tensor(). One AllGather.
  • Megatron with EP×TP×PP: parameters sharded along three axes. slime's bucket-wise weight update broadcasts within PP group, AllGathers within EP group, AllGathers within TP group, removes padding. Multi-step choreography.
  • DeepSpeed ZeRO-3: parameters live in CPU until materialized. Different lifecycle entirely.

A single method would either expose all this complexity to the user (defeating the abstraction) or hide it behind branching that grows with every backend (defeating the simplicity). Four explicit paths, each with clean semantics, win.

⑤ ZMQ for Engine, HTTP for Server

ZMQ is <1µs per message inter-process; HTTP is 100µs-1ms. For colocated runs where handle tuples cross processes thousands of times per second, ZMQ is mandatory. For disaggregated runs across a datacenter network, HTTP is mandatory — firewalls, retry semantics, statelessness. Same scheduler underneath. The framework picks the shell by topology.

⑥ Partial rollout vs strict on-policy

The math: if 10% of prompts are long-context (2K tokens decode) and 90% are short (512 tokens), the short ones finish in ~2s but the long ones take ~20s. Strict on-policy waits for all to finish: 90% of GPUs idle for 18s every iteration. Over 1000 iterations that's 5 GPU-hours per GPU wasted.

Partial rollout: train on the 128 that finished, let the rest continue under (now-stale) θₜ. Throughput 2-4× higher; off-policy ratio rises but TIS/MIS corrections (Miles) or curriculum scheduling (Kimi K1.5) bound the cost to <5% convergence slowdown.

Pitfalls — the 踩坑录 (lessons paid in pain)

Chenyang's tutorial calls out these specific traps. They're listed not for completeness but because each one cost the community real time.

1. Training-inference numerical drift ("BERT-era unsolved bug")

Inference kernels fuse operations to maximize throughput; the fusion depends on batch shape. The same model + same input + different batch size = slightly different output. For sampled tokens this is invisible. For logprobs that drive PPO loss, accumulated drift makes the policy appear to update toward a target that doesn't exist.

Lesson: Never trust inference-engine logprobs for loss computation. Always recompute with the training engine. Pay the cost.

2. Handle-tuple deserialization segfaults

verl's update_weights_from_tensor requires monkey_patch_torch_reductions() to register CUDA IPC handle deserializers. Missing this call → silent segfault on the SGLang side, but only intermittently (depends on which tensor's IPC handle lands first).

Lesson: Handle tuples are low-level. The abstraction leaks. Read the integration code line by line.

3. NCCL hang under mixed inference backends

OpenRLHF + SGLang integration: "the entire process was very painful... there's still an NCCL hang error that a DeepSpeed core contributor is currently fixing." NCCL hangs are silent deadlocks — no error, just a stalled run. Diagnosis requires NCCL_DEBUG=INFO, py-spy on every rank, and patience.

Lesson: Mixing distributed backends (DeepSpeed + Ray + SGLang's own dist) creates fragile NCCL group management. Prefer frameworks where one author owns the full stack.

4. Memory choreography under colocate

Megatron's CPU offload is "not perfect" — KV cache and model parameters contend for the same address ranges. slime's bucket-based weight update exists specifically to avoid OOM on large MoE models: instead of broadcasting all parameters then updating, broadcast in buckets that fit in spare memory.

Lesson: The "memory hand-off" between train and rollout phases is the most common production failure mode. Test at full scale early.

5. Off-policy ratio drift

Partial rollout introduces staleness. Without correction (TIS/MIS) or operational bounds (Kimi's curriculum), the off-policy ratio grows monotonically until the policy is training on data from a fundamentally different policy. The training curve looks fine. The eval curve mysteriously degrades.

Lesson: Monitor off-policy ratio as a first-class metric. Cap staleness even if it costs throughput.

6. Mixing precisions across train and infer

BF16 training + FP8 inference + FP32 optimizer state = three numerical regimes interacting via weight sync. MoE expert routing diverges between regimes (the problem Miles' R3 was built to solve). Symptom: training accuracy plateaus, eval gets worse, no obvious cause.

Lesson: If you can't make precision uniform, instrument routing decisions and gradients per-expert. The bug is invisible at the loss level.

The painful lesson behind all of these

Chenyang's most-repeated line, paraphrased: "A large number of RL conclusions derived from papers are based on RL infrastructure that may be extremely flawed." Every pitfall above produces results that look fine on the loss curve and fail silently on the policy. The discipline this demands — instrumenting routing, monitoring staleness, recomputing logprobs on the training engine — is what separates production RL from research toys.

SGLang RL Surface

Both Miles and SLIME — and SkyRL, RLinf, cosmos-rl optionally — use SGLang as the inference engine generating rollouts. The README explicitly calls SGLang "a proven rollout backend used for training many frontier models, with native RL integrations" — naming Miles, SLIME, AReaL, verl, and Tunix as adopters. This is the picks-and-shovels of the whole space.

Mental model. An RL trainer touches SGLang through three concerns: generate (rollouts), update weights (sync new policy after each step), and release memory (let training reclaim GPU during gradient updates).

The rollout cycle in code

# 1. Boot a server (or use Engine in-process)
from sglang import Engine
engine = Engine(model_path="Qwen/Qwen3-4B", tp_size=4)

# 2. Generate rollouts with logprobs (called every RL step)
out = engine.generate(
    prompts,
    sampling_params={"temperature": 1.0, "max_new_tokens": 1024},
    return_logprob=True,           # token-level probs for advantage / KL
)

# 3. After a training step, push new weights without restarting
engine.init_weights_update_group(...)
engine.update_weights_from_distributed(...)   # NCCL broadcast from trainer

# 4. Free GPU so trainer can use it for the next gradient step
engine.release_memory_occupation()
# ...trainer runs forward+backward...
engine.resume_memory_occupation()

Miles — bit-identical RL for frontier MoEs

Miles targets the hardest case: training trillion-parameter MoE models like DeepSeek V3/R1 where any numerical drift between inference and training collapses learning. Its three signature contributions — Unified FP8 Pipeline, R3 routing replay, and INT4 quant-aware training — all attack this single problem.

Architecture

  • Training: Megatron via miles_plugins/megatron_bridge/ (plugin-based, not direct integration)
  • Inference: SGLang with FP8/INT4 quantization patches
  • Orchestration: Ray
  • Algorithms: GRPO, PPO, GSPO, Reinforce++, DPO variants — all with TIS/MIS importance corrections

Why it matters

Without R3, the same input routes to different MoE experts in inference vs training because of FP8 precision differences. The gradient becomes noise. R3 captures the routing during SGLang generation and replays it during Megatron training. Combined with the unified FP8 pipeline, this gives bit-identical numerics — the only way to make MoE RL stable.

OpenRLHF — the OG, structurally limited

OpenRLHF was the first widely-adopted open-source RLHF framework. Chenyang's tutorial describes it as the predecessor to verl, with structural limitations that pushed the community toward newer designs.

Architecture

  • Training: DeepSpeed (primary), some FSDP
  • Inference: vLLM (primary), via --backend sglang for SGLang
  • Orchestration: Centralized Ray driver — single-controller all the way down
  • Algorithms: PPO and variants

Why it's structurally limited (per Chenyang)

The synchronous episode loop creates a hard sync point at broadcast_to_vllm() — training and inference share Ray actors, forcing sequential execution. DeepSpeed's ZeRO-3 memory efficiency is powerful, but the architecture sacrifices throughput for simplicity. Integrating SGLang causes torch dependency conflicts. NCCL hangs are a known unresolved pain.

Why it still matters

It's the easiest framework to read end-to-end. The Ray-actor pattern is unambiguous: one model per actor, one method per phase. Every modern framework's architecture is a response to OpenRLHF's structural choices — read it first to understand what later frameworks fixed.

AReaL — fully-async RL

AReaL (Inclusion AI; named explicitly in SGLang's README as an RL adopter) makes "fully asynchronous" its first-class design goal. While other frameworks pause training for weight sync, AReaL never blocks — the trainer queues weight updates and continues immediately.

Architecture

  • Training: FSDP or Megatron (both supported via separate engines)
  • Inference: SGLang (default — explicit in docs), vLLM optional
  • Orchestration: Ray + custom async future infrastructure
  • Distinctive primitive: _PendingWeightUpdateBucket — memory-bounded buckets of async NCCL broadcasts

The async secret

NCCL broadcasts are launched with async_op=True and bucketed by memory budget. The trainer queues bucket after bucket without ever blocking. Inference workers receive weight updates via futures — only inference pauses during the actual sync, not the trainer. Result: claimed 2.77× speedup over synchronous designs.

Trade-off

Off-policy ratio increases. AReaL's bet is that with careful staleness bounding and Kimi-K1.5-style curriculum scheduling, the off-policy cost stays under 5% convergence slowdown while delivering 2-3× throughput. Miles' IS corrections (TIS/MIS) would be the formal complement — explicit mathematical correction for what AReaL bounds operationally.

The TPU detour — Cambrian-MLLM

The Cambrian-MLLM TPU training blog documents two years of training multimodal models on Google TPUs. The systems-level findings are directly relevant to RL infra:

  • "Dynamic shapes are the enemy" — every shape change triggers XLA recompilation (minutes). RL pipelines on TPU must use fixed batch sizes and dummy-image padding. Variable-length generation as we know it on GPU is essentially banned.
  • "Arbitrary SPMD sharding proved impractical" — the team hit three critical bugs and fell back to "FSDP via SPMD." Ray-based hybrid placement doesn't translate.
  • Silent library incompatibilities — F.scaled_dot_product_attention and torch.utils.checkpoint fail without errors on TorchXLA.

The conclusion: a TPU-native RL framework would look fundamentally different. Static graph compilation, fixed shape contracts, no Ray, no SGLang. No one has built this yet. vLLM has plugin support for TPUs in inference, but the full RL-on-TPU story is open. This is a real research gap.

① The PPO step, written out — verl's fit() body

This is the structural backbone of every RL framework on this page. Every phase is a marked_timer block; the batch: DataProto accumulates fields via .union(...); the driver code is thin — all model compute happens inside dispatched method calls.

verl/trainer/ppo/ray_trainer.py· main training step · L1370+ 
with marked_timer("step", timing_raw):
    with marked_timer("gen", timing_raw, color="red"):
        combined_gen_output = self.async_rollout_manager.generate_sequences(combined_gen_batch)
        self.checkpoint_manager.sleep_replicas()

    batch = batch.repeat(repeat_times=self.config.actor_rollout_ref.rollout.n, interleave=True)
    batch = batch.union(gen_batch_output)

    with marked_timer("reward", timing_raw, color="yellow"):
        if self.use_rm and "rm_scores" not in batch.batch.keys():
            batch_reward = self._compute_reward_colocate(batch)
            batch = batch.union(batch_reward)
        reward_tensor, reward_extra_infos_dict = extract_reward(batch)

    with marked_timer("old_log_prob", timing_raw, color="blue"):
        old_log_prob, old_log_prob_mfu = self._compute_old_log_prob(batch)
        batch = batch.union(old_log_prob)

    if self.use_reference_policy:
        with marked_timer(str(Role.RefPolicy), timing_raw, color="olive"):
            ref_log_prob = self._compute_ref_log_prob(batch)
            batch = batch.union(ref_log_prob)

    if self.use_critic:
        with marked_timer("values", timing_raw, color="cyan"):
            values = self._compute_values(batch)
            batch = batch.union(values)

    with marked_timer("adv", timing_raw, color="brown"):
        batch = compute_advantage(
            batch, adv_estimator=self.config.algorithm.adv_estimator,
            gamma=self.config.algorithm.gamma, lam=self.config.algorithm.lam,
            num_repeat=self.config.actor_rollout_ref.rollout.n,
            norm_adv_by_std_in_grpo=norm_adv_by_std_in_grpo, config=self.config.algorithm,
        )

    if self.use_critic:
        with marked_timer("update_critic", timing_raw, color="pink"):
            critic_output = self._update_critic(batch)

    with marked_timer("update_actor", timing_raw, color="red"):
        actor_output = self._update_actor(batch)
What this teaches. The colors are not decorative — they correspond to phases you'd see in verl's wandb traces. Red = generation, yellow = reward, blue = old_log_prob, etc. The driver code is <30 lines for an entire PPO/GRPO/REINFORCE++ step because every method call is a Ray dispatch to SPMD worker groups. Reading this once is worth more than reading 20 papers.

② The four update_weights_from_* methods — SGLang's mixin

This is the verb that hides four topologies. Note that all four end with flush_cache_after_weight_update — the RadixCache must be invalidated because old prefix entries reference the old policy. The tensor/IPC variants add a torch.distributed.barrier for ordering.

python/sglang/srt/managers/scheduler_update_weights_mixin.py· four transports, one verb 
class SchedulerUpdateWeightsMixin:
    def update_weights_from_disk(self, recv_req):
        """In-place update of the weights from disk."""
        success, message = self.tp_worker.update_weights_from_disk(recv_req)
        if success and self.draft_worker is not None:
            success, message = self.draft_worker.update_weights_from_disk(recv_req)
        if success:
            self.flush_cache_after_weight_update(recv_req)
        return UpdateWeightFromDiskReqOutput(success, message, 0)

    def update_weights_from_distributed(self, recv_req):
        """Update the online model parameter."""
        success, message = self.tp_worker.update_weights_from_distributed(recv_req)
        if success:
            self.flush_cache_after_weight_update(recv_req)
        return UpdateWeightsFromDistributedReqOutput(success, message)

    def update_weights_from_tensor(self, recv_req):
        """Update the online model parameter from tensors."""
        worker = self.draft_worker or self.tp_worker  # supports speculative drafts
        success, message = worker.update_weights_from_tensor(recv_req)
        if success:
            self.flush_cache_after_weight_update(recv_req)
        torch.distributed.barrier(group=self.tp_cpu_group)
        return UpdateWeightsFromTensorReqOutput(success, message)

    def update_weights_from_ipc(self, recv_req):
        """Update the online model parameter from IPC for checkpoint-engine integration."""
        success, message = self.tp_worker.update_weights_from_ipc(recv_req)
        if success and self.draft_worker is not None:
            success, message = self.draft_worker.update_weights_from_ipc(recv_req)
        if success:
            self.flush_cache_after_weight_update(recv_req)
        torch.distributed.barrier(group=self.tp_cpu_group)
        return UpdateWeightsFromIPCReqOutput(success, message)
What this teaches. Same return-shape, same flush-cache, same draft-worker chain. The only difference between the four methods is what self.tp_worker.update_weights_from_* does internally — and that's where the topology-specific complexity lives. The mixin pattern means a framework above sees one verb; the worker below sees four distinct transports.

③ The memory choreography — release_memory_occupation

This is the single function that makes colocated RL economical. Three tag types (KV_CACHE, WEIGHTS, CUDA_GRAPH) map to three distinct GPU memory pools. Each can be paused independently.

python/sglang/srt/managers/scheduler_update_weights_mixin.py· tag-based memory pause 
def release_memory_occupation(self, recv_req):
    assert self.is_fully_idle(), \
        "release_memory_occupation should be called only when server is idle."

    tags = recv_req.tags or GPU_MEMORY_ALL_TYPES
    for tag in tags:
        self.offload_tags.add(tag)

    if GPU_MEMORY_TYPE_KV_CACHE in tags:
        self.memory_saver_adapter.pause(GPU_MEMORY_TYPE_KV_CACHE)
        self.flush_cache()

    if GPU_MEMORY_TYPE_WEIGHTS in tags:
        self.stashed_model_static_state = _export_static_state(
            self.tp_worker.model_runner.model
        )
        torch.distributed.barrier(self.tp_cpu_group)
        self.memory_saver_adapter.pause(GPU_MEMORY_TYPE_WEIGHTS)

    if GPU_MEMORY_TYPE_CUDA_GRAPH in tags:
        self.memory_saver_adapter.pause(GPU_MEMORY_TYPE_CUDA_GRAPH)

    torch.get_device_module().synchronize()
    return ReleaseMemoryOccupationReqOutput()
What this teaches. Notice the assert at the top: is_fully_idle(). This is the concurrency contract — the function literally cannot run if any request is in flight. The three independent if tag in tags blocks mean colocated training can free just KV cache (keep weights warm for fast resume) or just weights (the trainer needs the memory). Granularity is the whole point.

④ verl's DataProto — the universal data envelope

Every batch that flows between modules in verl is a DataProto. Three fields: batch (TensorDict for tensors), non_tensor_batch (numpy arrays for things like images), meta_info (Python dict for metadata). Indexing supports int, slice, list, numpy, and torch tensor — first-class data engineering.

verl/protocol.py· the universal data envelope 
@dataclass
class DataProto:
    """A DataProto is a data structure for data exchange between functions."""
    batch: TensorDict = None
    non_tensor_batch: dict = field(default_factory=dict)
    meta_info: dict = field(default_factory=dict)

    def __len__(self):
        if self.batch is not None:
            return self.batch.batch_size[0]
        elif self.non_tensor_batch is not None and len(self.non_tensor_batch) > 0:
            random_key = list(self.non_tensor_batch.keys())[0]
            return self.non_tensor_batch[random_key].shape[0]
        else:
            return 0

    def __getitem__(self, item):
        """Enhanced indexing supporting int, slice, list, numpy array, torch tensor."""
        if isinstance(item, slice):
            return self.slice(item.start, item.stop, item.step)
        elif isinstance(item, list | np.ndarray | torch.Tensor):
            return self.select_idxs(item)
        elif isinstance(item, int | np.integer):
            return self._get_item(item)
What this teaches. A clean data envelope is what allows the SPMD dispatch system to work. When you write batch = batch.union(gen_batch_output) in the fit loop, that union is well-defined because DataProto knows how to merge tensors with tensors and dicts with dicts. Without this abstraction, the fit loop becomes a swamp of dict-merging boilerplate.

⑤ verl's dispatch decorator — SPMD made invisible

The driver writes self.actor_rollout_wg.update_actor(batch). That single line becomes: split batch across worker ranks → remote-call each worker → gather results → concatenate. The decorator does all of it.

def dispatch_dp_compute_data_proto(worker_group, *args, **kwargs):

What this teaches. "Single-controller interface, multi-controller execution." The framework looks single-threaded; it runs SPMD across hundreds of GPUs. Auto-padding handles the case where batch size isn't divisible by world_size. This pattern (dispatch / execute / collect) is the universal RL workflow primitive — RLinf's M2Flow generalizes it.

The difference between "synchronous" and "fully async" RL is one keyword: async_op=True. The trainer launches broadcasts and immediately continues.

⑦ OpenRLHF's synchronous contrast

OpenRLHF's broadcast_to_vllm blocks until all engines acknowledge. The trainer can't start the next step until inference has the new weights.

openrlhf/trainer/ray/ppo_actor.py· the structural limit Chenyang identified 
@ray.remote(num_gpus=1)
class PolicyModelActor(BaseModelActor):
    def init_model_from_pretrained(self, ...):
        self.model = actor_model
        self.trainer = ActorPPOTrainer(self.model, ...)

    def fit(self, kl_ctl):
        return self.trainer.ppo_train(kl_ctl)

    def broadcast_to_vllm(self, vllm_engines):
        # synchronous: waits for all engines to acknowledge
        for engine in vllm_engines:
            engine.update_weights.remote(...)
        # blocks here until done
What this teaches. The structural limit Chenyang identified. OpenRLHF's design is correct but synchronous. Each step pays a fixed weight-sync latency before the next rollout can begin. At 256 GPUs with frequent updates, that latency cost grows from a nuisance to the dominant term in step time. AReaL and verl's async paths exist because of this limit.

⑧ The RadixCache lifecycle — reference-counted prefix sharing

Inside SGLang's KV cache lives the radix tree that makes GRPO group rollouts efficient. The key methods (signatures from radix_cache.py):

python/sglang/srt/mem_cache/radix_cache.py· reference-counted prefix tree 
class RadixCache(BasePrefixCache, KVCacheEventMixin):
    def match_prefix(self, params: MatchPrefixParams) -> MatchResult:
        """Walk the tree, return the longest matching prefix's KV slots."""

    def insert(self, params: InsertParams) -> InsertResult:
        """Add a new completion's tail tokens as descendant nodes."""

    def inc_lock_ref(self, node: TreeNode) -> IncLockRefResult:
        """Pin a node — its KV slots cannot be evicted while lock_ref > 0."""

    def dec_lock_ref(self, ...) -> DecLockRefResult:
        """Release pin — node becomes evictable when no requests reference it."""

    def evict(self, params: EvictParams) -> EvictResult:
        """LRU/LFU/SLRU eviction of evictable nodes only — pinned nodes survive."""

    @property
    def evictable_size(self):
        """How much memory can we reclaim if we evict everything unpinned?"""
What this teaches. Six methods, automatic memory safety. inc_lock_ref pins a node while a request uses it. The eviction policy (LRU/LFU/SLRU/FIFO/FILO/MRU/Priority — all implemented in evict_policy.py) only touches unpinned nodes. Use-after-free is unrepresentable — a node with lock_ref > 0 simply cannot be evicted. This is correctness-by-construction in 50 lines of method signatures.

The meta-pattern across all eight excerpts

Notice what you didn't see in any of these snippets: explicit locking, manual memory management, error recovery for partial state. The patterns make those concerns disappear by structure. marked_timer blocks scope every phase. DataProto.union merges types cleanly. flush_cache_after_weight_update appears in every weight-update path. is_fully_idle() asserts the precondition. inc_lock_ref pins what can't be freed. This is what "engineered for correctness" looks like at the line level — not paranoia about bugs, but architectures where the buggy thing can't be written.

Getting started — a learning path

If I were learning this space from scratch today, I'd walk it like this:

  1. Week 1 — RL primitives. Sutton & Barto Ch. 13, OpenAI Spinning Up, GRPO paper. Don't touch any framework yet.
  2. Week 2 — SGLang. It's the substrate. Read entrypoints/engine.py, scheduler_update_weights_mixin.py, radix_attention.py. Understand the four update_weights_from_* flavors.
  3. Week 3 — One simple framework end-to-end. SLIME's geo3k_vlm_single_turn or SkyRL's examples/train/gsm8k. Run it. Read every file train.py imports.
  4. Week 4 — Trace the cycle. In whatever you ran, find the exact line where generate() is called, where the reward is computed, where update_weights_from_distributed is called. These are the three pillars in the wild.
  5. Week 5+ — Pick a specialization. Numerics → Miles. Agents → SkyRL. Embodied → RLinf. World models → cosmos-rl. Now the deep features make sense because you have the universal frame.
